# Code-Change Discipline — алгоритм работы с существующим кодом

## Когда применять

Перед любым изменением кода, который ты НЕ написал в этой сессии: рефакторинг, багфикс, расширение функциональности, сопровождение, читать-чтобы-понять-перед-другой-задачей.

## Зачем это существует

Большинство багов вносится в момент, когда автор изменения уверенно делает допущение о существующем коде, не проверяя его. Этот алгоритм — нормативный guard против семи самых частых ошибок такого рода. Подкрепляющие принципы (FPF) объясняют, ПОЧЕМУ guard работает.

Алгоритм усиливает FC-принципы:
- **2. Минимальные изменения** — без понимания идеи нельзя «минимально» её менять.
- **16. Сначала понять задачу** — здесь применяется к существующему коду, а не к новой задаче.
- **19. Ruthless Deletion of Unused Code** — но НЕ за счёт удаления информации (см. шаг 7).

## 7 шагов

1. **Понять идею** — зачем код написан так, а не иначе. Не «как работает», а «какую задачу решает». Если непонятно — исследовать тестами.
2. **Поставить идею под сомнение первой** — смена идеи приводит к самым большим изменениям. Если идея верная — менять реализацию. Если кривая — менять идею.
3. **Определить свои допущения** — где я додумал, на основе чего. Каждое допущение — гипотеза, требующая проверки.
4. **Проверять гипотезы кодом** — написать тест, запустить, посмотреть. Не утверждать «ведёт себя так» без доказательства.
5. **Спрашивать человека** там, где не уверен как система ДОЛЖНА себя вести (поведение vs идея).
6. **Не менять контракт метода** без явного обсуждения — сигнатура, возвращаемый тип, поведение для вызывающего кода = breaking change.
7. **Не убирать информацию** — если лог был шумный но информативный, заменить его на лучший, а не удалять.

## Подкрепляющие принципы (FPF — First Principles Framework)

### Strict Distinction (FPF A.7) — различай роль и реализацию

Не путай что метод ДЕЛАЕТ (его контракт/роль) с тем КАК он реализован.

> "Provide a single, didactically clear lattice of distinctions that keeps models free from category errors."

Три ключевых различия:
- **Role vs Function** — роль (контракт, позиция) это НЕ поведение. Если метод возвращает response — это его роль, менять возвращаемый тип = менять контракт.
- **MethodDescription vs Method vs Work** — описание (как ДОЛЖНО работать) ≠ capability (что МОЖЕТ делать) ≠ occurrence (что РЕАЛЬНО произошло). Нельзя утверждать «ведёт себя так» (Work) на основании чтения кода (MethodDescription) — нужен запуск (evidence).
- **Category collapse** — когда пишут «function/role/process» взаимозаменяемо, команда перестаёт понимать, меняют ли они план, capability или фиксируют факт.

**Normative guard:** Если два метода ВЫГЛЯДЯТ похоже, но имеют разные контракты (сигнатура, возвращаемый тип, побочные эффекты) — это разные роли. Паттерн из одного нельзя переносить в другой без понимания различий.

### Evidence Graph (FPF A.10) — утверждение без доказательства = мнение

> "A claim without a chain is only an opinion."

Правило: каждое утверждение о поведении кода должно быть подкреплено evidence (тест, запуск, лог).

Пять failure modes без evidence:
1. **Weightless claims** — «этот метод работает так же» без теста = утверждение без носителя.
2. **Collapsed scopes** — смешивание «что задумано» (design-time) и «что происходит» (run-time). Нельзя читать код и делать выводы о поведении — нужен запуск.
3. **Self-justifying loops** — обосновывать изменение тем же изменением («я добавил guard, значит теперь правильно»).
4. **Source loss** — при рефакторинге теряется контекст: зачем код был написан так, кто автор, какой issue.
5. **Temporal ambiguity** — «раньше работало так» без evidence из конкретной версии.

**Normative guard:** Перед утверждением «X ведёт себя как Y» — написать тест, запустить, показать результат. Prediction → Run → Compare.

### Ontological Parsimony (FPF A.11) — добавляй только то, что нельзя вычесть

> "Add only what you cannot subtract."

Протокол минимальной достаточности:
1. **Composition** — можно ли выразить изменение через существующие примитивы без потери смысла?
2. **Non-Redundancy** — не дублирует ли изменение то, что уже есть?
3. **Functional Naming** — имя описывает ЧТО делает, а не из чего состоит?
4. **Sharp Boundary** — можно ли одним предложением отличить, когда применять, а когда нет?

**Normative guard:** Предпочитай минимальное точечное изменение. НЕ рефакторинг метода, НЕ смена сигнатуры, НЕ перенос логики. Если можно обойтись добавлением нескольких строк вместо переписывания метода — добавь несколько строк.

### BoundedContext (FPF A.1.1) — смысл локален

> "Make meaning local; make translation explicit."

У каждого модуля/метода свой семантический контекст — набор инвариантов, контракт, словарь.

> "Absent an explicit semantic frame: ambiguity becomes structural debt, pluralism looks like contradiction, local rules leak globally."

**Normative guard:** Нельзя переносить паттерн из одного контекста в другой без понимания: какие инварианты различаются, какой контракт у каждого, почему автор написал иначе. «Одинаковый код» в разных контекстах может иметь РАЗНЫЙ смысл.
